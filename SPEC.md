# SPEC.md — 天津麻将

## 一、问题陈述

**要解决的问题**：天津麻将是一种具有独特规则体系的地方麻将变种（混儿机制、无吃无点炮、仅自摸胡），但目前缺乏一个专注于天津规则的、界面现代简洁的在线麻将游戏。主流麻将平台多为国标/四川/广东等大众规则，天津本地玩家缺乏一款规则正宗、上手即玩的轻量级麻将游戏。

**目标用户**：喜欢天津麻将规则的麻将爱好者，希望随时在浏览器中开一局对战 AI。

**为什么值得做**：天津麻将规则独特且有趣，规则复杂度适中（约 10 种番型、混儿机制），非常适合作为 AI4SE 课程项目——既能体现规约驱动的完整流程，又有足够的逻辑复杂度来检验智能体的编程能力。

## 二、用户故事

### US1：创建对局
> 作为一个玩家，我希望打开页面后能快速选择人数（2-4 人）并开始一局天津麻将，这样我不用等待就能开始游戏。

**验收**：页面加载后，选择人数，点击开始，3 秒内进入对局界面。

### US2：行牌操作
> 作为一个玩家，我希望在自己的回合摸牌后，能从手牌中选择一张打出，并能清晰地看到其他玩家的弃牌和碰/杠操作，这样我能正常进行游戏。

**验收**：点击手牌中的非混儿牌可打出；其他人碰/杠时界面有明显反馈；轮到 AI 时自动执行。

### US3：碰与杠
> 作为一个玩家，当别人打出的牌与我手牌匹配时，我希望能选择碰或杠，并执行对应操作。

**验收**：满足条件时弹出碰/杠按钮，点击后执行操作并更新界面。杠后自动补牌。

### US4：听牌提示
> 作为一个玩家，当我的手牌接近胡牌时，我希望能看到听牌提示（还差什么牌），这样我能更好地规划出牌策略。

**验收**：手牌达到听牌状态时，界面显示"听牌"标识及等待的牌。

### US5：胡牌判定
> 作为一个玩家，当我摸到能组成 4 刻子 + 1 对子的牌时，我希望游戏自动判定我自摸胡牌并展示结果。

**验收**：摸到胡牌时自动触发胡牌判定，展示所有玩家手牌和番型名称。

### US6：断线恢复
> 作为一个玩家，当我不小心刷新页面或关闭浏览器后，我希望重新打开时能恢复到之前的对局状态。

**验收**：刷新页面或重新打开后，自动恢复牌局（手牌、弃牌、当前回合玩家等）。

### US7：再来一局
> 作为一个玩家，一局结束后我希望点击"再来一局"按钮快速开始下一局，且庄家按规则轮转。

**验收**：结算后点击"再来一局"，自动重新发牌，庄家按规则轮转（胡牌者连庄，否则逆时针）。

## 三、功能规约

### 3.1 对局管理

| 功能 | 输入 | 行为 | 输出 | 边界/错误 |
|------|------|------|------|-----------|
| 创建对局 | 人数 n (2-4) | 初始化游戏状态，随机定庄，洗牌，翻混儿，发牌 | 游戏状态对象 | n 不在 2-4 范围拒绝 |
| 恢复对局 | 无（自动检测 localStorage） | 读取已保存的游戏状态并恢复 | 游戏状态对象 | 无存档时显示新建界面 |
| 结束对局 | 自摸胡或流局 | 展示结算，记录结果 | 结算信息 | — |

### 3.2 行牌逻辑

| 功能 | 输入 | 行为 | 输出 | 边界/错误 |
|------|------|------|------|-----------|
| 摸牌 | 当前玩家 | 从牌墙头部取一张牌 | 摸到的牌 | 牌墙空则流局 |
| 出牌 | 玩家选择的牌 | 将该牌移入弃牌区，设为当前弃牌 | 更新后的游戏状态 | 混儿不能打出；非自己回合不能出牌 |
| 碰 | 当前弃牌 | 玩家手牌中 2 张 + 弃牌组成刻子，展示在碰区 | 更新后的状态 | 手中无对子则不可碰 |
| 杠（明杠） | 当前弃牌 | 手牌中 3 张 + 弃牌组成杠，补牌 | 更新后的状态 | 条件不满足不可杠 |
| 杠（暗杠） | 手中 4 张相同 | 4 张组成暗杠，补牌 | 更新后的状态 | — |
| 杠（补杠） | 已碰刻子 + 手牌第 4 张 | 扩为杠，补牌 | 更新后的状态 | 需有已碰刻子 |
| 金杠 | 手中 4 张混儿 | 4 张混儿组成暗杠，补牌 | 更新后的状态 | — |

### 3.3 混儿

| 功能 | 输入 | 行为 | 输出 | 边界 |
|------|------|------|------|------|
| 翻混儿 | 牌墙 | 取最后一张翻出为指示牌；该牌剩余 3 张 + 同花色下一序列 4 张 = 7 张混儿 | 7 张混儿标记 | 序列循环：万/条/筒/风/箭各自独立 |
| 混儿打出 | 玩家试图打出混儿 | **拒绝** | 提示"混儿不能打出" | — |

### 3.4 胡牌判定

| 功能 | 输入 | 行为 | 输出 | 边界 |
|------|------|------|------|------|
| 自摸判定 | 摸到的牌 + 手牌 | 检查是否组成 4 刻子 + 1 对子 | 布尔值 + 番型名称 | v1 仅支持 4 刻子 + 1 对子 |

### 3.5 庄家轮转

| 规则 | 条件 |
|------|------|
| 首局 | 随机指定庄家 |
| 胡牌者连庄 | 本局胡牌者下局继续坐庄 |
| 逆时针轮转 | 流局或他人胡 → 庄家逆时针传给下家 |

### 3.6 AI

| 功能 | 行为 |
|------|------|
| 摸牌后出牌 | 从手牌中随机选择一张非混儿牌打出 |
| 碰/杠判断 | 满足条件时随机决定是否碰/杠 |

### 3.7 胡牌牌型系统架构

牌型判定采用**插件式架构**：
- 每个牌型独立为一个模块，实现统一接口 `(hand, context) => PatternResult | null`
- 通过注册表管理启用哪些牌型
- 新增牌型：创建文件 + 注册，不修改核心引擎

v1 仅注册一个牌型：**4 刻子 + 1 对子**。混儿的万能牌作用、更多牌型在后续迭代中添加。

### 3.8 前端界面

- 游戏桌面：显示各玩家手牌（己方可见全部，AI 仅示背面）、弃牌区、碰/杠区
- 操作提示：碰/杠可用时显示操作按钮
- 听牌状态：显示"听牌"标识
- 结算弹窗：胡牌时展示各玩家手牌 + 番型名称
- 新建对局：选择人数界面

| 维度 | 要求 |
|------|------|

### 5.3 数据流

```
[用户点击"开始"] → POST /api/games → 服务端初始化状态 → 返回给前端

[轮到玩家] → GET /api/games/:id/actions → 获取可执行操作
           → 前端 @tj-mahjong/shared 计算提示（听牌、可碰可杠）

[玩家出牌] → POST /api/games/:id/actions { type: "discard", tile }
           → 服务端 shared 校验合法性 → 更新状态 → 返回新状态

[AI回合] → 服务端调用 shared/ai.ts → 随机出牌 → 更新状态 → 返回新状态

[状态变化] → 前端存 localStorage
```

## 六、数据模型

### 6.1 核心类型

```typescript
type Suit = 'wan' | 'tiao' | 'tong' | 'feng' | 'jian';

interface Tile {
  suit: Suit;
  rank: number;       // 万条筒: 1-9; 风: 1-4(东南西北); 箭: 1-3(中发白)
  id: number;         // 0-135 唯一标识
  isHun: boolean;     // 是否为混儿
}

interface Player {
  seat: number;       // 0-3, 0=庄家
  hand: Tile[];       // 手牌 (已排序)
  discards: Tile[];   // 弃牌区
  melds: Meld[];      // 碰/杠区
  isHuman: boolean;
}

type MeldType = 'pong' | 'ming-kong' | 'an-kong' | 'bu-kong' | 'jin-kong';

interface Meld {
  type: MeldType;
  tiles: Tile[];
  source?: number;    // 来源玩家 seat（明杠/碰时）
}

type GamePhase = 'setup' | 'playing' | 'finished';

interface GameState {
  id: string;
  players: Player[];
  wall: Tile[];          // 剩余牌墙
  hunIndicator: Tile;    // 混儿指示牌
  currentSeat: number;
  phase: GamePhase;
  lastDiscard: Tile | null;
  lastDiscardSeat: number | null;
  winner: number | null;
  winPattern: string | null;
}
```

### 6.2 牌型插件接口

```typescript
interface PatternChecker {
  name: string;
  check(hand: Tile[], context: { hunTiles: Tile[] }): PatternResult | null;
}

interface PatternResult {
  patternName: string;
  description: string;
}
```

## 七、API 设计

### 7.1 端点列表

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/games` | 创建新对局 |
| GET | `/api/games/:id` | 获取对局状态 |
| GET | `/api/games/:id/actions` | 获取当前可执行操作 |
| POST | `/api/games/:id/actions` | 提交操作 |

### 7.2 请求/响应

**POST /api/games**
```json
// Request
{ "playerCount": 4 }
// Response 201
{ "game": { "id": "...", "state": { ...GameState } } }
```

**GET /api/games/:id**
```json
// Response 200
{ "game": { "id": "...", "state": { ...GameState } } }
// Response 404
{ "error": "GAME_NOT_FOUND" }
```

**GET /api/games/:id/actions**
```json
// Response 200
{
  "canPong": true,
  "canMingKong": false,
  "canAnKong": false,
  "canBuKong": false,
  "canWin": true,
  "legalDiscards": [0, 1, 3, 5, 7, 8, 9, 10, 11, 12]  // 可出的牌索引（混儿不在内）
}
```

**POST /api/games/:id/actions**
```json
// 出牌
{ "type": "discard", "tileIndex": 3 }
// 碰
{ "type": "pong" }
// 杠
{ "type": "kong", "kongType": "ming-kong" }
// 胡
{ "type": "win" }

// Response 200
{ "game": { "id": "...", "state": { ...GameState } } }
// Response 400
{ "error": "INVALID_ACTION", "message": "混儿不能打出" }
```

## 八、技术选型与理由

| 层 | 技术 | 理由 |
|----|------|------|
| 前端框架 | React 18 + TypeScript | Open Design 原生适配 React；生态最丰富 |
| 构建工具 | Vite | 开发 HMR 极快，构建产物小 |
| UI 样式 | Tailwind CSS | 快速构建现代简约风格，与 Open Design 配合良好 |
| 设计系统 | **Open Design — Linear 风格** | Linear 黑白灰极简基调，干净利落，符合"现代简约"方向；无冗余装饰，适合游戏界面；71 个系统中与目标风格最匹配 |
| Open Design Skill | `web-app` | 通用 Web 应用模板，最接近游戏界面需求（非 landing/dashboard） |
| 后端框架 | Express + TypeScript | 轻量、与前端同语言、生态成熟 |
| 数据库 | SQLite (better-sqlite3) | 零配置、零依赖、嵌入式，适合单机游戏场景 |
| 包管理 | pnpm workspace | Monorepo 原生支持，安装快，磁盘省 |
| 测试 | Vitest | 与 Vite 共用配置，TypeScript 原生支持，速度快 |
| 类型检查 | TypeScript strict mode | 减少运行时错误，提升 subagent 输出质量 |
| 代码风格 | ESLint + Prettier | 统一风格 |
| 容器化 | Docker + docker-compose | 作业要求，单命令启动 |
| CI | GitHub Actions | 每次 push 跑测试 + 构建镜像 |

## 九、验收标准

### 9.1 对局流程

- [ ] 可创建 2/3/4 人对局
- [ ] 开局自动洗牌、翻混儿（7 张正确）、发牌

## 十、风险与未决问题

### 风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 天津麻将规则理解偏差 | 游戏逻辑不符合真实规则 | 与用户持续确认规则细节（已记录于 SPEC_PROCESS.md） |
| 胡牌牌型插件架构过度设计 | 开发周期延长 | v1 只注册 1 个牌型，架构先搭好不做复杂抽象 |
| AI 随机出牌体验差 | 游戏可玩性低 | 约定 v1 不追求 AI 质量，后续迭代升级 |
| Monorepo 配置复杂 | subagent 环境搭建耗时 | 使用 pnpm workspace 标准模板，最小化配置 |

### 未决问题

| 问题 | 状态 |
|------|------|
| 混儿作为万能牌的胡牌判定逻辑 | 推迟到 v2 |
| 番型具体列表与判定规则 | 推迟到 v2（v1 仅 4 刻子 + 1 对子） |
| 游戏速度调节 | v1 不做 |
| 音效 | v1 不做 |
| 计分系统 | v1 不做 |
| 联机功能 | 未来扩展 |
