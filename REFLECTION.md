> 本文由 Claude (DeepSeek-v4-pro) 辅助撰写与润色，核心观点与经验教训来自项目实际过程。

---

## 一、哪些 Superpowers 技能发挥了最大作用？

### 真正有价值的

**brainstorming（不可替代）**

如果没有 brainstorming 的逐轮追问机制，我大概率会带着对天津麻将的模糊理解直接写代码。结果就是：会实现"标准麻将+天津特色"，而不是"天津麻将"。实际过程中 8 轮追问产出了 9 项规则修正——混儿从 4 张纠正到 7 张、会儿体系被完全取消、吃操作被移除、点炮被移除。这些修正如果发生在写代码阶段，返工成本会高一个数量级。

brainstorming 最聪明的设计是 **"一次只问一个问题"**。这看似低效，实则避免了"问三个问题只得到一个答案"的信息丢失。

**subagent-driven-development（效率放大器）**

8 个 subagent 独立完成了从类型定义到 Express 服务端的全部代码，每个都在 2-5 分钟内交付，且附带了完整测试。194 个测试全部通过，这个是实打实的效率提升。

**writing-plans（关键桥梁）**

PLAN.md 把 SPEC.md 中的自然语言需求翻译成了 subagent 能直接执行的 task 列表。每个 task 有明确的文件路径、依赖关系、测试要求。没有 PLAN.md，subagent 派发就是一场赌博。

### 形式大于实质的

**requesting-code-review / receiving-code-review**

两阶段评审（spec 合规 + 代码质量）在理论上很好，但在实际执行中被弱化了。原因是：task 粒度过小（一个 task 通常只涉及 1-2 个文件），spec 合规检查基本等价于"测试是否通过"。代码质量检查在没有人工 review 的情况下，agent 自审的深度有限。

**finishing-a-development-branch**

合入 master 的操作被简化为 `git merge`。在实际项目中这步涉及 PR、CI 检查、冲突解决——但本项目是单人开发，没有真正的 code review 流程，finishing 技能沦为形式。

## 二、TDD 强制在 AI 协作下的体感

### 是放大器，不是阻碍

每个 subagent 都被要求先写失败的测试（RED），再写最小实现（GREEN），最后重构。实际效果：

- **正向**：168 个 shared 测试 + 26 个 server 测试 = 194 个测试，一次跑过。这是 TDD 最大的价值——测试不是"事后补"，而是"定义完成的标准"。
- **正向**：subagent 在写实现之前被迫思考"这个函数输入什么、输出什么"，减少了方向性错误。
- **中性**：部分测试过于琐碎（如 `types.test.ts` 中的联合类型验证），对代码质量的贡献有限。

### 一个未解决的问题

TDD 在**跨 task 集成**层面缺失。每个 task 的测试都孤立地通过了，但没有集成测试验证"server 调 shared 引擎的完整链路"。这更像是 PLAN.md 的问题而非 TDD 方法论的问题——plan 中没有包含集成测试 task。

## 三、Subagent 工作流的自主性与粒度

### 能跑多久不偏题？

在本项目中，subagent 的"自主运行距离"大约是 **2-5 分钟 / 1 个 task**。超过这个范围（比如把 Task 9 的 server 实现拆成多个子 task），agent 容易在中间某个步骤产生偏差，且偏差会累积。

### 什么样的颗粒度最优？

基于 8 次 subagent 派发的经验：

| 粒度 | 例子 | 效果 |
|------|------|------|
| **太细**（< 1 分钟） | "写一个类型定义" | 过度拆分，commit 过多 |
| **最优**（2-5 分钟） | "实现 tile.ts 和全部测试" | agent 可以完整执行 TDD 循环，不会跑偏 |
| **太粗**（> 10 分钟） | "实现整个客户端"（Tasks 10-12 合并） | agent 可能在某些组件上偷懒，质量不均匀 |

最优粒度是：**1 个文件 + 1 个测试文件的完整 TDD 循环**。

## 四、SPEC 与 PLAN 质量对实现质量的影响

### 正向案例

SPEC.md 中对 `GameState` 的数据模型定义非常精确——直接成为了 `types.ts` 的蓝图。subagent 只需要照搬接口定义，零歧义。

### 负面案例

混儿规则在 SPEC.md 中只写了"7 张混儿 = 指示牌剩余 3 张 + 同花色下一序列 4 张"，但没有明确"风牌和箭牌的序列独立循环"。幸运的是在 brainstorming 阶段已经充分讨论过，subagent 实现的 `tile.ts` 中有完整的 `getNextTileInSequence` 逻辑。

### 核心教训

SPEC.md 的**例子密度**决定了 subagent 的正确率。有具体例子的规则（如"翻五万 → 3 张五万 + 4 张六万 = 7 张混儿"），agent 几乎不会出错。只有抽象描述的规则（如"序列循环"），agent 有概率做错。

## 五、最有效的 Prompt / Context 策略

### 上下文截断策略

Subagent 不需要完整的 SPEC.md 和 PLAN.md。在派发时，我只提供：
1. **前置依赖**（哪些文件已存在，导出了什么函数）
2. **本 task 的具体要求**（输入/输出/边界条件）
3. **测试要求**（具体到测试用例）

这个策略让 subagent 的上下文保持精简，避免信息过载。

### 类型系统作为"活文档"

TypeScript 的类型定义是 subagent 最好的 guide。`Tile`、`GameState`、`PlayerAction` 这些类型本身就是精确的 spec——agent 不需要猜测参数类型，编译器会告诉它对错。

### 有效策略总结

| 策略 | 效果 |
|------|------|
| 提供前置依赖的函数签名 | 减少 agent 查找代码的时间 |
| 给出具体测试用例 | agent 不需要自己设计测试 |
| 明确禁止项（"不能做 X"） | 比"请做 Y"更能防止跑偏 |
| 提供 commit message 模板 | 确保格式统一 |

## 六、Open Design 的使用情况

坦白说，本项目**没有实质性地使用 Open Design**。在 brainstorming 阶段选定了 Linear 设计系统 + web-app skill，但在客户端实现时，subagent 直接使用了 Tailwind CSS 的 utility class 来构建界面，没有调用 Open Design 的设计系统组件或遵循 Linear 的设计 token。

原因有三：
1. Open Design 的 skill 库偏向 landing/dashboard/blog 等业务场景，与麻将游戏的 UI 需求不匹配
2. 前端 subagent 对 Tailwind CSS 的熟练度高于 Open Design 的特定组件库
3. 在 v1 阶段，"能用"的优先级远高于"好看"

如果重新选择，对于一个**游戏项目**，Open Design 是否合适本身就是个值得讨论的问题。它的 71 个设计系统都是企业级 SaaS 风格，与游戏的视觉调性天然冲突。强制使用 Open Design 可能反而不如直接用 Tailwind CSS 手写更符合"现代简约"的设计目标。

## 七、如果重做，会改变什么

1. **先写规则文档，再 brainstorming**。60% 的 brainstorming 时间花在了规则澄清上。如果先整理一份天津麻将规则文档（参考维基百科或麻将 Wiki），然后让 AI 对照检查，效率会高得多。

2. **在 shared 引擎完成时就做集成测试**。不要等到 server 和 client 都写完了才验证"前后端能通"。

3. **PLAN.md 中的代码示例应该更少、更抽象**。subagent 有时会逐字照搬 plan 中的代码，失去了独立判断。更好的做法是给出接口定义 + 测试用例，让 agent 自己设计实现。

4. **client 和 server 应该配端到端测试**。当前只有 unit test，没有验证"创建游戏 → 发牌 → 玩家出牌 → AI 出牌 → 胡牌"的完整链路。

5. **混儿万能牌逻辑应该在 v1 就做**。当前 v1 只生成了混儿但不使用，导致"听牌判定"和"胡牌判定"都不完整——game loop 能跑，但游戏性不足。

## 八、对 Superpowers 方法论的批判

### Superpowers 假设了什么？

1. **假设 spec 是正确的**。整个工作流（brainstorming → plan → implement）的前提是 spec 没有错误。但在 novelty 较高的项目（如特定规则的游戏）中，spec 很可能存在对规则的误解，而这种误解会沿着工作流一路传递到代码中。

2. **假设 task 可以独立执行**。subagent-driven-development 的前提是 task 之间无共享状态。但实际开发中，后一个 task 的实现方式常常取决于前一个 task 的具体实现细节（而不只是接口），这种耦合在 plan 阶段很难完全预见。

3. **假设 TDD 能保证正确性**。单元测试覆盖了函数行为，但没有覆盖"函数组合后的行为"。这在游戏项目中尤其明显——每个操作（摸/出/碰/杠/胡）单独正确，但组合后的游戏流程可能仍然有 bug。

### 这些假设在本项目中成立吗？

部分成立。前两个假设基本成立（spec 经过多轮修正比较准确，task 拆分粒度合理）。第三个假设不完全成立——我们确实没有验证完整的游戏流程。

### Superpowers 最大的弱点

**对"人类判断"的依赖被低估了**。Superpowers 声称"让你把精力聚焦在真正属于人类的判断：架构、产品、质量"。但实际体验是：你需要做的判断远不止这些——spec 的准确性、task 的拆分粒度、subagent 产出的合理性、什么时候跳过某个流程——这些判断贯穿始终且没有自动化替代方案。

## 九、对当前 AI4SE 工具与方法论的整体看法

### 智能体编码的现实

经过这个项目，我对 AI 辅助编码的认识是：**AI 可以承担大约 70% 的编码工作量，但需要人类承担 100% 的设计判断和约 40% 的调试工作**。这里的"调试"不是找 bug，而是"理解 AI 为什么这样写、判断这样写对不对、在不对的时候指导它改正"。

### 超级智能体 vs 方法论的张力

Superpowers 试图用流程来约束智能体的行为——这本质上是一个"用方法论弥补模型缺陷"的思路。随着模型能力的提升（GPT-5、Claude 5），部分流程可能被内化为模型能力。例如"先红再绿"的 TDD 流程，未来可能模型能自己判断"需要先写测试"而不需要外部技能强制。

### AI4SE 课程的核心价值

这门课最有价值的不是 Superpowers 本身，而是建立了一种**与 AI 协作的工程纪律**。在 AI 能写大部分代码的时代，区分一个工程师和一个只会写 prompt 的用户的，正是这种纪律：**spec 要经得起陌生人的冷启动、code review 不能偷懒、测试不能事后补**。
