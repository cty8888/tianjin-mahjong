### 1.3 规则修正记录

#### 修正 1：混儿确定规则（关键修正）

**AI 初始理解**（错误）：翻到一张牌，该牌的下一张为混儿，共 4 张。

**用户纠正**：
- 翻一张牌作为**指示牌**
- 指示牌剩下的 **3 张** + 下一序列的 **4 张** = 共 **7 张混儿**
- 指示牌本身不算混儿

**花色序列**（各自独立循环）：

| 花色 | 序列 |
|------|------|
| 万 | 一万 → 二万 → … → 九万 → 一万 |
| 条 | 一条 → 二条 → … → 九条 → 一条 |
| 筒 | 一筒 → 二筒 → … → 九筒 → 一筒 |
| 风牌 | 东 → 南 → 西 → 北 → 东 |
| 箭牌 | 中 → 发 → 白 → 中 |

**举例**：翻到五万 → 3 张五万 + 4 张六万 = 7 张混儿

#### 修正 2：取消会儿体系

**AI 初始理解**（错误）：游戏有"会儿"（二级混儿），会会儿杠、本会儿龙等番型。

**用户纠正**：只有混儿，没有会儿。混儿仅开局翻一次（7 张），整局不变。

**影响**：移除以下番型 — 会儿杠、本会儿龙。保留 — 素本混儿、混儿杠、杠开、混儿吊、捉五魁、龙、本混儿龙。

#### 修正 3：无"吃"操作

**AI 初始假设**（错误）：标准麻将规则，包含吃（Chi）。

**用户纠正**：天津麻将只有**碰和杠**，没有吃。

**影响**：去除所有"吃"相关逻辑。他人回合响应优先级：杠 > 碰。所有面子必须是对子（碰形成刻子）或杠，无法通过顺子成型。

#### 修正 4：无点炮，只能自摸

**AI 初始假设**（错误）：标准麻将，点炮和自摸均可胡牌。

**用户纠正**：天津麻将**只能自摸胡牌**，不能胡别人打出的牌。

**影响**：胡牌只在自己的回合摸牌后判定，他人回合不涉及胡牌响应。

#### 修正 5：杠后只补牌，不翻混儿

**AI 初始假设**（错误）：每次杠后翻新一组混儿（7 张），累计叠加。

**用户纠正**：杠后只补一张牌（从牌墙末尾），**不翻新混儿**。

**影响**：混儿只在开局翻一次（7 张），整局不变。删除所有"杠后翻混儿"相关逻辑。

#### 修正 6：杠的分类与金杠

**杠的类型**：

| 类型 | 条件 |
|------|------|
| 明杠 | 手中有三张，别人打出第四张 |
| 暗杠 | 自己摸到四张相同（含金杠） |
| 补杠 | 已碰过的刻子，自己摸到第四张 |

**金杠**：四张混儿组成的暗杠（注意：混儿共 7 张，其中 4 张为同一花色数字的"下一序列牌"可凑成金杠）。

#### 修正 8：无"报听"

**用户纠正**：天津麻将没有报听（立直），听牌不需要声明。

#### 修正 9：胡牌牌型架构要求

**用户需求**：胡牌牌型的判定逻辑必须是**可配置的**，方便添加/删除牌型。不是硬编码。

**架构影响**：shared 引擎需要设计一个**插件式牌型系统**：
- 每个番型/牌型独立成模块
- 统一接口：输入 GameState + PlayerHand → 输出是否匹配 + 番型名称
- 通过注册表/配置文件控制启用哪些牌型
- 新增牌型只需创建新文件 + 注册，不修改核心引擎

## 五、反思

### brainstorming 技能评价

**做得好：**
- 逐轮追问的设计确实有助于收敛需求。在 8 轮追问中，每一轮只问一个问题，避免了信息过载
- "先问平台、再问技术栈、再问规则"的递进顺序合乎逻辑
- 多种"修正"机制让用户有反复纠正的空间（规则修正共 9 项），这是 brainstorming 最有价值的部分

**做得不好：**
- 初始对天津麻将规则的了解不足，在混儿/会儿/吃/点炮等关键规则上反复出错，增加了沟通成本
- 技术选型阶段没有让用户参与足够多的决策（用户说了"由你推荐"），但实际上技术栈选择对开发体验影响很大
- 没有在某些关键节点（如"确定混儿是 7 张"之后）主动做规则总结，导致后续仍出现了杠后翻混儿的错误假设

### 关于 Superpowers 方法论

Superpowers 的 7 步工作流在本项目中发挥了约束作用，但并非所有技能都同等重要：

- **brainstorming**：不可替代。9 项规则修正都是在对话中完成的，如果跳过这步直接写代码，代价会非常大
- **writing-plans**：产出的 PLAN.md 对 subagent 派发至关重要。但 PLAN.md 中的代码示例有些过于详细，接近"手把手教"，在实际 subagent 执行时，agent 有时会忽略 plan 中的具体实现走自己的路
- **subagent-driven-development**：确实提升了效率。8 个 subagent 独立完成了 8 个 task，每个都在 1-5 分钟内完成，且都有完整的测试
- **test-driven-development**：在 subagent 层面执行得不错——每个 task 都是先写测试、再实现。但在集成层面缺少端到端测试（前后端联调尚未验证）

### 最大教训

1. **规则澄清的成本远高于预期**。天津麻将的规则修正占了 brainstorming 约 60% 的对话量。如果一开始先让用户写一份规则文档，效率会高得多
2. **subagent 产出的代码风格不完全一致**。虽然都通过了测试，但不同 subagent 的命名习惯、注释密度、函数拆分粒度有差异
3. **PLAN.md 中的代码示例与实际产出有偏差**。subagent 有时会用自己的实现替代 plan 中的代码，但这不一定是坏事——有时 agent 的实现更好
